# Implementation Plan: deployd-api durability chain

**Branch**: `145-deployd-durability` | **Date**: 2026-05-10 | **Spec**: [`spec.md`](./spec.md)

## Summary

Land four coupled changes that together make the
`deployments` / `deployment_events` audit trail in deployd-api-rs
survive pod eviction, fresh-PVC pods, and cluster-wide disasters:
flip the Hetzner chart's `persistence.enabled` to `true`, narrow the
container-start `rm` so it does not erase application data, enable
`backup`+`s3`+`auto-heal` on the Hiqlite dep with operator-supplied
S3 credentials wired through env, and add restore-on-startup so a
pod against an empty volume rehydrates from the most recent encrypted
snapshot before becoming Ready. A runbook accompanies the change.

The load-bearing realignments:

1. **One PR ships all four edges.** Splitting them lands a half-fix:
   the chart change without the Cargo features makes the volume
   durable but the data path still has no DR story; the Cargo
   features without the chart change make the application aware of
   S3 backup but the volume gets wiped at every restart.
2. **Operator owns secrets and cadence.** Cron schedule, retention,
   S3 endpoint, bucket, and `cryptr` encryption key are
   operator-supplied via Helm values + a Kubernetes Secret. The
   service code reads them through a typed config struct.
3. **Restore-on-startup is opt-in via configuration.** A dev cluster
   with no `BackupConfig` populated continues to start with an empty
   data dir; only configured clusters block readiness on a successful
   restore.
4. **Spec-code coupling is intentionally cross-layer.** The
   `implements:` list spans chart values, chart templates, Rust
   manifest+lockfile, Rust service code, and a runbook doc. Spec 130's
   primary-owner heuristic places primary ownership on spec 145 since
   this is the spec that motivates the coupling.

## Technical Context

**Languages**: Rust (`deployd-api-rs`, axum + Hiqlite, separate Cargo
workspace from `crates/`), Helm/YAML (deployd-api chart). Markdown for
the runbook.
**Primary Dependencies**: `hiqlite ~0.13` (resolves to `0.13.1`);
`cryptr` and `s3-simple` already present transitively in
`platform/services/deployd-api-rs/Cargo.lock`. ExternalSecrets
operator (already deployed) for projecting the operator-controlled
secret store into the pod.
**Storage**: PVC-backed Hiqlite data dir on `/var/lib/deployd/data`;
S3-compatible object store for encrypted snapshots (operator-supplied
endpoint and bucket).
**Testing**: Rust unit/integration tests against the new BackupConfig
+ restore path (where feasible without a live S3); chart `helm
template` smoke tests; live cluster validation against the Hetzner
deploy for AC-1 / AC-2 / AC-4 (pod-eviction and fresh-PVC scenarios).
**Target Platform**: Hetzner block-storage cluster (primary); managed
K8s targets (Azure / AWS / GCP / DO) inherit the chart default.
**Project Type**: Cross-layer platform-delivery feature — chart +
service code + ops doc.
**Performance Goals**: NFR-001 (restore-on-startup ≤ 60s for
steady-state); NFR-005 (no new latency on the steady-state pod-restart
path).
**Constraints**:
- No new direct deps (FR-007).
- No password / secret material in `values.yaml` (NFR-003).
- No silent fall-back to empty-database start when restore is
  configured (FR-006).
- The `implements:` list is the authoritative coupling surface;
  spec 127 + 130 + 133 accept it.
**Scale/Scope**: Today's `deployments` table is small (single-active-org
deploy history, tens of rows); `deployment_events` is append-only and
likewise small at current scale. NFR-001's 60s bound is generous for
today's corpus and the implementation should beat it comfortably.

## Phase 0 — Pre-flight investigation

**Purpose**: Resolve any remaining open questions in `spec.md` §6 and
confirm assumptions before writing code. Operator decisions resolved
2026-05-10 (Q1 storage, Q3 cron, Q4 cryptr, Q5 env files) are
recorded in `spec.md` §6 "Resolved" — Phase 0 verifies them rather
than re-litigating.

P0.1 **Verify Hetzner storage class + size against the live cluster.**
Decision (resolved): `size: 1Gi`, `storageClassName: hcloud-volumes`.
Verification: the hcloud CSI driver enforces a minimum (commonly
10 GiB). Run `kubectl get pvc -n <ns>` against the existing
deployd-api workload (or `hcloud volume list`) to confirm 1 GiB is
acceptable, OR observe the actual size the provisioner allocates
when 1 GiB is requested. If rejected/rounded, AMEND `spec.md` §2.1
+ §6 to the actual minimum the driver accepts (likely 10 GiB) and
update the values file accordingly. **Halt before editing manifests
if 1 GiB is not viable.**

behaviour after an unclean shutdown — does it still leave a stale
`state_machine/lock`? If yes → Option A (narrow `rm -f` of just the
lock file). If no → Option B (drop the wrapper shell entirely). The
test plan: bring up a single-pod Hiqlite cluster against a PVC, send
`SIGKILL`, restart pod, observe whether init succeeds without
intervention. (Implementation-time decision; not an operator ask.)

P0.3 **Cron schedule + retention.** Decision (resolved):
`schedule: "0 */6 * * *"`, `keep: 28`. Chart-level default in
`values.yaml`; NFR-002 keeps both operator-configurable per env.

## Phase 1 — Cargo + restore-on-startup

**Purpose**: enable Hiqlite features and wire `NodeConfig` for backup
+ restore.

**Sequence:**

1. **Edit `platform/services/deployd-api-rs/Cargo.toml:17`** to:
   ```toml
   hiqlite = { version = "~0.13", default-features = false, features = ["sqlite", "backup", "s3", "auto-heal"] }
   ```
2. **Regenerate `platform/services/deployd-api-rs/Cargo.lock`** via
   `cargo build --manifest-path platform/services/deployd-api-rs/Cargo.toml`.
   Inspect the diff: `cron` should appear; `cryptr`, `s3-simple`,
   `deadpool`, `rusqlite` already present. Halt if any direct dep
   appears or if the diff is wider than feature activation.
3. **Add `BackupConfig` struct to `src/config.rs`** (or create the
   file if it does not exist; verify against current source). Fields:
   `s3_endpoint`, `s3_bucket`, `s3_access_key`, `s3_secret_key`,
   `s3_path_prefix` (Option), `cryptr_key`, `cron_schedule`,
   `keep_count`. Loader reads from env (`DEPLOYD_BACKUP_*` prefix),
   returns `None` when no env keys are set (opt-in).
4. **Wire `NodeConfig.backup_config`** in `src/store.rs::init_db`
   (lines 13-33). When `BackupConfig::from_env()` returns `Some`,
   populate the upstream config struct; when `None`, leave the
   default (no backup).
5. **Implement restore-on-startup** in `src/main.rs:24-28` (or
   `src/store.rs::init_db` if upstream supports it inline). Algorithm
   per §2.4: inspect data dir → if empty + BackupConfig → restore
   from latest snapshot → fail-fast on restore error. Readiness
   probe stays not-Ready until `init_db` returns Ok.
6. **Tests:**
   - `tests/config_test.rs` (or inline `#[cfg(test)]` in
     `src/config.rs`): `BackupConfig::from_env` honours all env vars,
     returns `None` when no vars present, errors when partial config
     (e.g. endpoint set but key missing).
   - `tests/store_test.rs`: `init_db` against a temp dir without
     BackupConfig — equivalent to current behaviour.
   - `tests/restore_test.rs`: skipped under `#[ignore]` unless an
     S3-compatible test endpoint is configured (e.g. localstack or
     minio). Verifies restore against a known snapshot when run.

**Phase 1 exit criterion:** `cargo build / check / clippy / test
--manifest-path platform/services/deployd-api-rs/Cargo.toml` is
green; the new BackupConfig struct loads correctly from env; restore
path executes without panic on the empty-config (None) path.

## Phase 4 — Live validation + spec close

**Purpose**: verify acceptance criteria on the actual Hetzner deploy,
refresh registries, mark spec implementation complete.

> **Hands-on ownership note.** Steps 2–5 below are live-cluster ops
> (`kubectl delete pod`, `kubectl delete pvc`, real S3 writes). The
> implementation agent prepares commands and surfaces diagnostics;
> the operator drives the destructive operations and acks each AC
> validation. Auto mode does not authorise autonomous destructive
> ops against a live cluster. Each `kubectl delete ...` step is a
> halt-and-confirm boundary.

**Sequence:**

1. **Deploy to Hetzner** (`make deploy-hetzner` or the equivalent
   chart-apply path). Confirm pod comes up cleanly.
2. **AC-1: pod eviction.** Insert a known synthetic deploy-event row
   (or pick an existing one from a recent deploy). `kubectl delete
   pod -n <ns> deployd-api-...`. Wait for replacement pod Ready.
   Re-query — row present.
3. **AC-2: fresh-PVC restore.** `kubectl delete pvc` (after
   confirming a backup snapshot is in S3). Force pod restart. Watch
   readiness probe — pod stays NotReady through the restore window.
   Once Ready, query — rowset matches the most recent snapshot.
4. **AC-3: cron snapshot emission.** Wait one cron cycle (or trigger
   manually if the upstream supports it). Verify a new encrypted
   object appears in S3 at the configured prefix. Decrypt with the
   cryptr key on a workstation as a smoke check.
5. **AC-4: scrub no longer deletes data.** Pod restart against a
   populated PVC; confirm `deployments` and `deployment_events`
   rowsets are unchanged.
6. **Run `make ci` (warm)** — exit 0.
7. **Run the coupling check**:
   `./tools/spec-spine/spec-code-coupling-check/target/release/spec-code-coupling-check`
   → no warnings against spec 145's `implements:` list. (AC-8.)
8. **Recompile spec registry + codebase index.**
9. **Update spec 145 frontmatter:** `implementation: complete`,
   `closed: "<today>"`. Recompile registry. Confirm
   `registry-consumer status-report` reflects the change.
10. **Open PR.** Title:
    `feat(spec-145): deployd-api durability chain — PVC + scrub-narrow + s3 backup + restore-on-startup`.

**Phase 4 exit criterion:** AC-1 through AC-9 in `spec.md` §3.3
pass; PR open.

## Test strategy

- **Unit (Rust):** `BackupConfig::from_env` parsing + validation;
  `init_db` paths under (no config) and (config with empty data dir,
  with restore stubbed).
- **Integration (Rust, `#[ignore]` by default):** end-to-end restore
  against a localstack/minio test endpoint. Run manually pre-merge
  on at least one developer workstation.
- **Helm:** `helm template -f values-hetzner.yaml` (and one of the
  other env files) renders without error; rendered Deployment carries
  expected fields.
- **Live (Hetzner):** AC-1 through AC-4 validated against the running
  cluster. This is the load-bearing validation; unit tests cannot
  cover real PVC behaviour or real S3 round-trips.

## Risks and rollback

- **Restore failure on a real cluster.** If the upstream restore API
  is more delicate than P0.6 anticipated, revert §2.4 only — leave
  the chart edits, `backup`+`s3`+`auto-heal` enablement, and cron
  emission active. The cluster is safer (PVC durable, snapshots in
  S3) even without restore-on-startup wired; AC-2 fails but AC-1 /
  AC-3 / AC-4 pass. AMEND the spec to defer §2.4 to a follow-up,
  rather than holding the partial fix back.
- **Hetzner storage-class drift.** If the resolved `storageClassName`
  is wrong, the Pod will block on Pending PVC. Roll back values-file
  edit; re-attempt with the operator-confirmed value.
- **Cron schedule misconfiguration.** Hiqlite's cron parser may
  reject formats the operator expects. Document in the runbook the
  exact format Hiqlite v0.13.1 accepts (typically a 5- or 6-field
  cron expression).
- **Lockfile surprise.** If `cron` brings in a transitive that
  conflicts with another deployd-api-rs dep, reconcile in Phase 1.
  Most likely a no-op given the audit-confirmed transitives.
- **Operator-secret rollout.** Adding new keys to the operator-side
  secret store is a manual op step; if the secret is not present
  when the chart upgrade applies, the pod will fail readiness on the
  missing-config path. Document the apply-order in the runbook
  (secret keys first, then `helm upgrade`).

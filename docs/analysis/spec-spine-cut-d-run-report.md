### W-03 / W-05 — ImplementsField polymorphism

- Production registry.json carries three shapes for `implements:`
  (bare spec-id scalar; list-of-`{path}` objects; empty list). W-03's
  initial enum modelled only Scalar + List<String>; W-05 surfaced
  the gap on the live corpus. Bundled fix in **W-05**: widened
  enum to `Scalar(String) | List(Vec<Value>)` with `paths()` and
  `as_scalar()` helpers, and corrected the spec 147 semantic
  (scalar form is a spec-id, NOT a path → `paths()` returns empty
  for Scalar). The W-03 test for `paths()` was updated in W-05 to
  match the corrected semantic. This is the one place where a
  prior PR's test changed in a later PR; documented in the W-05
  commit body.

### W-06c — `compliance` retained in KNOWN_KEYS

- Deviation from plan letter; refined factoring documented in-code.
  The plan's literal instruction was to drop `compliance` from
  `KNOWN_KEYS` in shared-types. Implementing it caused
  `spec-compiler compile` to fail on the live corpus with 8 V-002
  errors (compliance is a complex YAML mapping that
  `extra_frontmatter` can't represent). The plan's *intent* (no
  compliance field in registry.json output) was already satisfied
  by removing the `FeatureRecord.compliance` field. KNOWN_KEYS is
  now factored as a "permitted frontmatter" allowlist distinct from
  "fields spec-compiler emits" — a defensible refinement of the
  plan's mechanism that preserves the plan's intent. Re-added
  `"compliance"` to KNOWN_KEYS with a doc comment explaining the
  new factoring.

# AIM Vue + Node Scaffold

This directory is a pointer, not a code repository. The authoritative scaffold lives upstream at:

- **Remote**: `GovAlta-Pronghorn/template`
- **Local clone**: `~/Dev2/template` (this repo is a sibling, not a submodule)

## How the Factory Uses the Scaffold

The Factory engine copies the upstream scaffold into the target project directory before dispatching the adapter-handoff phase. The engine resolves the scaffold source in this order:

1. `--scaffold-source <path>` CLI flag (highest precedence — useful for local development against an unpublished branch).
2. `adapter.scaffold.source` in `adapter-manifest.yaml` (points at the upstream remote + ref).
3. Fallback: the last-synced revision captured by the `/factory-sync` command (see `factory/upstream-map.yaml` `template.last_synced_sha`).

## Entry Points in the Upstream Template

The template ships two top-level scripts that the Factory dispatches after the scaffold is copied:

| Script                       | Variant                           | Purpose                                                                                         |
| ---------------------------- | --------------------------------- | ----------------------------------------------------------------------------------------------- |
| `scripts/setup-app.ts`       | `single-public`, `single-internal`, `minimal` | Single-app scaffold. Copies base, selects web variant, installs modules per profile.        |
| `scripts/setup-dual-app.ts`  | `dual`                            | Dual-app scaffold. Produces separate `apps/api-public`, `apps/api-internal`, `apps/web-public`, `apps/web-internal` stacks. |

Both scripts accept the flags the adapter manifest's `args_schema` declares: `--profile`, `--dest`, `--yes`, `--dry-run`, `--clean`, `--no-install`.

## Module System

The template is module-based. Each capability (auth driver, session store, data connector, security middleware, user management) ships under `modules/*/` and is installed by `scripts/add-module.ts`. The factory does not hand-edit `apps/api/src/modules.ts` — it is regenerated by the module installer from `template.json`.

The adapter manifest's `scaffold.modules` block declares which modules ship for each variant profile. Those names MUST match `modules/*/manifest.json.name` upstream. When the template adds a new module, update the adapter manifest in the same PR that bumps `template.last_synced_sha`.

## Node Version

The template requires Node 24.x. Stagecraft's Create path executes the scaffold scripts on a Node-24 runner only — adapters declaring a different `scaffold.runtime` are not Create-eligible via the web UI today (spec 112 §5.2). For those, projects enter the platform via Import of an already-executed repository.

## Post-Scaffold Handoff

After the setup script completes:

1. The factory engine writes `.factory/pipeline-state.json` at L0 seed per spec 112 §5.2 step 5.
2. Stagecraft creates the GitHub repo and pushes commit #1 containing the scaffold + L0 pipeline state.
3. Post-push execution moves to OPC. Stagecraft drops its working copy.

The invariant is: **birth on stagecraft, life on OPC**. Any adapter-specific work after commit #1 — ACP engine runs, reconcile actions, re-extractions — happens in OPC cockpit, never back on stagecraft.

## Not Bundled Here

- The template's source code (`apps/`, `packages/`, `modules/`, `scripts/`, `docker/`) is not checked into OAP. It is cloned fresh during sync / scaffold.
- `CODEMAP.md` and `PLACEHOLDERS.md` from the upstream are not mirrored into this directory; the adapter manifest references them by upstream path.

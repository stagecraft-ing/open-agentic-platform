# Prisma Migration Pattern

Prisma Migrate generates SQL migrations from schema changes. Migrations
are created with `npx prisma migrate dev` and stored in `prisma/migrations/`.

## Convention

- Migrations auto-generated by Prisma from schema diffs
- Each migration is a directory: `prisma/migrations/{timestamp}_{name}/migration.sql`
- Run `npx prisma migrate dev --name {name}` to create
- Run `npx prisma migrate deploy` in production

## Template

The migration SQL is generated by Prisma. For reference, a typical migration:

```sql
-- CreateEnum
CREATE TYPE "entity_status" AS ENUM ('DRAFT', 'SUBMITTED', 'APPROVED');

-- CreateTable
CREATE TABLE "entity" (
    "id" UUID NOT NULL DEFAULT gen_random_uuid(),
    "field_name" TEXT NOT NULL,
    "ref_id" UUID NOT NULL,
    "status" "entity_status" NOT NULL DEFAULT 'DRAFT',
    "created_at" TIMESTAMPTZ NOT NULL DEFAULT CURRENT_TIMESTAMP,
    "updated_at" TIMESTAMPTZ NOT NULL,

    CONSTRAINT "entity_pkey" PRIMARY KEY ("id")
);

-- CreateIndex
CREATE INDEX "entity_ref_id_idx" ON "entity"("ref_id");

-- AddForeignKey
ALTER TABLE "entity" ADD CONSTRAINT "entity_ref_id_fkey"
    FOREIGN KEY ("ref_id") REFERENCES "ref_entity"("id") ON DELETE CASCADE ON UPDATE CASCADE;
```

## Workflow

```
1. Edit prisma/schema.prisma
2. Run: npx prisma migrate dev --name add_entity
3. Prisma generates migration SQL in prisma/migrations/
4. Prisma applies the migration to the dev database
5. Prisma regenerates the Prisma Client
```

## Rules

1. Never hand-edit generated migration SQL unless absolutely necessary.
2. Create migrations in dependency order — referenced tables before referencing.
3. Name migrations descriptively: `add_sites`, `add_checks_to_sites`.
4. Run `npx prisma generate` after any schema change to update the client.
5. Check migration SQL into version control.
6. Use `npx prisma migrate deploy` in CI/production — not `migrate dev`.

# From Prompt to Pipeline: The Factory Evolution

## Executive Summary

> **Historical note:** `goa-software-factory` (GovAlta-Pronghorn/goa-software-factory) was historically tracked as `the_factory`; `template` (GovAlta-Pronghorn/template) was historically tracked as `AIM-vue-node-template`. The current names are used throughout this document.

The original factory system (`goa-software-factory`) was a pioneering attempt at AI-driven software delivery for a provincial government client. It proved that an AI agent could transform raw business documents into a complete, standards-compliant web application across five structured stages. A real dual-stack government application with SAML + Entra ID authentication, 11 API controllers, 24 route modules, and full public/staff Vue 3 frontends was generated by this system.

But the system had a fundamental architectural flaw: **it used the AI's context window as both the execution runtime and the state store.** Every design decision; the 44KB monolithic orchestrator, the manual file uploads, the operator confirmation gates typed into chat, the 12 Execution Discipline rules that Claude was asked to self-enforce; was a workaround for the fact that there was no engine outside the model. The prompt *was* the pipeline.

The new factory replaces this with a three-layer, contract-first architecture driven by a Rust execution engine. A single CLI command:

```bash
factory-run \
      --adapter aim-vue-node \
      --project ~/.factory-proof/example-project \
      --business-docs ~/.factory-proof/example-project.build-spec.yaml \
      --factory-root factory \
      --org example-org \
      --auto-approve \
      --scaffold-source ./upstream/template \
      --model opus \
      --extended-context \
      --thinking max \
      --step-timeout 1200
```

replaces: a Claude Projects chat session, two manual folder uploads, a copy-pasted activation prompt, six hand-typed stage confirmations, an operator guide with 20+ manual steps, and an open-ended post-generation debugging cycle.

This document traces the evolution, identifies the core design failures, and catalogues every capability gained.

## 1. The Original System

### What goa-software-factory Was

`goa-software-factory` was a collection of markdown files; approximately 70 documents totalling over 100,000 tokens; that encoded the complete knowledge of how to build a government client web application. When uploaded to a Claude Projects session and activated with a carefully constructed prompt, Claude would read the orchestrator file, execute five stages in sequence, and produce a complete application.

The system had two subsystems:

- **Factory Agent**; a 5-stage construction pipeline (Business Requirements → Service Requirements → Database Design → API Code → Client Interface), coordinated by a master orchestrator (`factory-orchestration.md`) that dispatched to 9 sub-orchestrator files and approximately 30 specialized skill files.
- **Security Agent**; a defensive (Blue Team) and offensive (Red Team) security assessment suite, with 12+ skill files for ASVS compliance, CAS rule verification, threat modelling, and automated scanning.

### How It Was Operated

The operator workflow, documented in the Pronghorn Operator Guide v3, was entirely manual:

1. Clone `goa-software-factory` to a local path (e.g., `C:\tools\goa-software-factory`)
2. Create a project from the AIM template via a web portal (Template Distributor)
3. Collect business documents (Word files, PDFs, meeting notes) into an `artifacts/` folder
4. Open claude.ai, create a Claude Project
5. Upload the entire `goa-software-factory/` folder to the Project
6. Upload the `artifacts/` folder
7. Compose an activation prompt with database connection string and schema name
8. Send the prompt and wait for Stage 1 output
9. Read the Handoff Report, perform a "sniff test", type confirmation to proceed
10. Repeat for Stages 2–5
11. After generation, run `npm install && npm test`, paste failures back to Claude, iterate
12. Configure `.env`, run migrations, verify tables in pgAdmin
13. Create a feature branch, commit, push, create a PR, wait for team lead approval
14. Trigger GitHub Actions deployment manually

### The AIM Template

The template was a production-quality monorepo scaffold: Vue 3 + Express 5 + a government design system, with 13 installable modules (auth drivers, session stores, database connectors, API gateway), a variant system (public/internal/dual), 21 architecture invariants, and its own 1,840-line orchestration layer. When `goa-software-factory` detected a `template-orchestrator.md` in the project root, it switched to "Template Mode"; generating a Build Specification (what to build) and handing it to the template's orchestrator (how to build it).

The template was sophisticated and well-engineered. The problem was never the template. The problem was how the factory consumed it.

## 2. The Core Problem

### The Context Window as Runtime

Every architectural decision in goa-software-factory traces back to a single root cause: **the AI model's context window was the only execution environment.** There was no engine, no state machine, no verification harness, no durable storage outside of what Claude could hold in memory and write to disk via tool calls.

This created six cascading failures:

#### 2.1 Monolithic Instruction Payload

The master orchestrator (`factory-orchestration.md`) contained: 12 Execution Discipline rules, a working state JSON schema, a pipeline Mermaid diagram, a selective-read artifact table, pre-flight checks, and a stage dispatch table; all in one 44KB file. It was split into 9 sub-files not for modularity, but because it exceeded practical context limits as a single document. Each sub-file remained hundreds of lines of dense instruction.

The total instruction payload for a factory run; orchestrator + stage skills + sub-agent skills + reference files + page-type specs with HTML samples; exceeded 100,000 tokens. Claude had to simultaneously be a requirements analyst, database architect, API developer, frontend developer, test writer, and self-validator, all while tracking which of 12 Execution Discipline rules it was supposed to be following.

#### 2.2 Self-Enforced Rules

The 12 Execution Discipline rules and the `execution_contract: binding: mandatory, enforcement: strict, on_violation: halt_and_reread` blocks in every skill file were instructions to Claude, not machine-enforced constraints. They were *wishes expressed as directives*.

The Blue Team security agent's `CLAUDE.md` documented over 20 "confirmed real-world failure modes"; each one a rule that Claude had violated in production. The response to each violation was to add another rule. The instruction payload grew; the probability of any single rule being followed decreased; more rules were added.

This is the fundamental impossibility: a system where the agent both executes and polices itself cannot guarantee compliance. The enforcement loop has no external authority.

#### 2.3 Ephemeral State

Pipeline state lived in Claude's context window. The working-state.json and stage validation logs were written to disk, but the orchestration state; which stage was current, which artifacts had been validated, what decisions had been made; existed only in the conversation. If the session timed out, if Claude hit its compute budget, if the operator accidentally started a new chat, all state was lost.

The Operator Guide acknowledged this directly:

> "Claude has a compute-based session limit on a 5-hour rotating window. If you hit the limit, wait for the window to reset before continuing; do not start a new project, just resume the same conversation."

If context was lost, the only recovery was to restart the entire pipeline.

#### 2.4 Manual Context Passing

Artifacts moved between stages via the conversation context, not via files. The orchestrator's "selective-read artifact table" was an instruction telling Claude which sections of which upstream files to read before each stage; an elaborate workaround for the fact that loading all artifacts simultaneously would overflow the context window. If Claude failed to read the right sections, downstream stages operated on incomplete information.

Between the operator's machine and Claude's context, file transfer was manual: drag-and-drop folder upload to the Claude Projects UI. There was no programmatic handoff, no checksumming, no validation that all required files were present.

#### 2.5 Batch Generation Without Verification

The factory generated the entire application in five sequential passes; all requirements, then all database DDL, then all API controllers, then all UI pages. There was no per-feature verification. The operator's only checkpoint was the "sniff test"; read the Handoff Report and decide if it "looks correct."

A schema misalignment introduced in Stage 3 would propagate silently through Stages 4 and 5. A missing API endpoint would not be discovered until the UI tried to call it. An auth middleware misconfiguration would not surface until the application was actually started. All integration errors accumulated silently until the post-generation debugging phase.

#### 2.6 Client-Specific Coupling

The process layer was inseparable from the implementation layer. The orchestrator referenced government design system web components, government cybersecurity standards, protected-level threat models, government ministry RACI structures, and government deployment infrastructure throughout. Every stage skill file contained client-specific compliance checks (FAC-Sn-xxx rules) that were meaningful only in that government context.

Using the same factory for a different tech stack; React instead of Vue, Next.js instead of Express, PostgreSQL instead of SQL Server; required rewriting dozens of files across both the factory and the template.

## 3. Evidence: What Went Wrong in Practice

### The Reference Application

A real dual-stack government application generated by goa-software-factory is the most complete evidence of both the system's capabilities and its failure modes.

**What the factory produced:** A structurally complete dual-stack application with:
- `apps/api-public`; Express REST API with SAML 2.0 auth, Redis sessions
- `apps/api-internal`; Express REST API with Entra ID auth, PostgreSQL sessions
- `apps/web-public`; Vue 3 citizen-facing SPA
- `apps/web-internal`; Vue 3 staff-facing SPA
- 11 controllers, 24 route modules, domain services for funding requests, assessments, decisions, organizations, programs, budgets, documents, community data
- Specialized services: Infomart data ingestion, blob storage, notifications

**What happened after generation:** The git history records at least 15 fix commits after the initial "All 5 stages complete" commit:

| Issue | Root Cause | Stage Where Introduced |
|-------|-----------|----------------------|
| Auth redirect falling back to localhost in production | Middleware misconfiguration | Stage 4 |
| Express 5 wildcard array gateway proxy bug | Framework-specific API misuse | Stage 4 |
| DDL schema misalignment in `audit_entry` INSERTs | Column name mismatch between DDL and service code | Stage 3 → Stage 4 |
| CSRF bypass needed for BFF gateway requests | Cross-cutting security concern not propagated | Stage 4 |
| Mock auth and role derivation issues | Auth driver contract violation | Stage 4 |
| 500 errors in community-data controller | Query/schema column mismatch | Stage 3 → Stage 4 |
| 404 on web-internal from BFF gateway prefix | URL path resolution across dual stacks | Stage 4 → Stage 5 |
| SPA routes returning 404 in Azure | `NODE_ENV` not set in deployment config | Stage 4 (server config) |
| Design system component type/form submission bugs | Design system component API misuse | Stage 5 |
| TypeScript typecheck errors across monorepo | Type contracts violated between packages | Stage 4 + Stage 5 |

Every one of these issues would have been caught by a `compile + test` verification gate after the individual feature was generated. Instead, they accumulated silently through five batch-generation stages and were discovered only during manual post-generation debugging; an open-ended process with no bound on iterations.

### The Operator Guide as Symptom

The Pronghorn Operator Guide v3 (April 6, 2026) is itself evidence of the problem. It is a 20+ step manual procedure targeting operators with "no developer background assumed"; yet the post-generation phase requires reading stack traces, understanding lint errors, reasoning about database schemas, and iterating on code failures with Claude.

The guide embeds individual names as infrastructure dependencies ("Your infrastructure team (Lingjia, Cody) typically handles this setup"), documents the session limit as an operational hazard, asks operators to paste database passwords in plaintext into Claude chat ("If your connection string contains a password, treat this message as sensitive"), and instructs them to perform a "sniff test" on machine-generated artifacts they may not be qualified to evaluate.

This is not a criticism of the guide's authors; it is a recognition that the manual coordination overhead was the best achievable outcome given the architectural constraints. The guide was a human-process patch for the absence of a machine execution layer.

## 4. The New Architecture

### Three-Layer Separation

The new factory decomposes the monolithic prompt into three layers with formal contracts between them:

```
┌─────────────────────────────────────────────────────┐
│  Process Layer (factory/process/)                    │
│  Universal, technology-free pipeline                 │
│  7 stages × 1 focused agent each                    │
│  Produces: Build Specification                       │
├──────────────── Contract Layer ─────────────────────┤
│  Build Spec · Adapter Manifest · Verification ·     │
│  Pipeline State                                      │
│  YAML schemas with Rust types + JSON Schema checks   │
├─────────────────────────────────────────────────────┤
│  Adapter Layer (factory/adapters/{name}/)            │
│  Pluggable tech stacks                               │
│  aim-vue-node · next-prisma · encore-react ·         │
│  rust-axum                                           │
│  Each: manifest + agents + patterns + invariants     │
└─────────────────────────────────────────────────────┘
```

### Rust Execution Engine

The engine (`crates/factory-engine/`) is not a prompt. It is compiled Rust code that:

- Discovers and validates adapters at startup (`AdapterRegistry::discover`)
- Generates typed workflow manifests for both phases
- Dispatches agents via the OAP orchestrator with bounded context
- Runs verification gates after every step using real commands (`npm run build`, `npm test`, `npm run lint`)
- Maintains durable pipeline state in `.factory/pipeline-state.json`
- Topologically sorts entity generation by foreign-key dependencies
- Generates scoped policy bundles for the permission runtime
- Supports crash recovery via `--resume <uuid>`

### Typed Contract Layer

The four contract schemas are defined in YAML (`factory/contract/schemas/`) and have corresponding Rust types (`crates/factory-contracts/`). The Build Specification; the central artifact that describes what to build; is validated by JSON Schema at pre-flight, parsed into typed Rust structs during phase transition, and hashed (SHA-256) for integrity verification.

This means the contract between "what to build" and "how to build it" is machine-enforced, not instruction-enforced.

## 5. The CLI: One Command, Full Pipeline

### Before: 20+ Manual Steps

```
1.  Clone goa-software-factory to C:\tools\goa-software-factory
2.  Create project via Template Distributor web portal
3.  Collect business documents into artifacts/ folder
4.  Open claude.ai → Create Claude Project
5.  Upload goa-software-factory/ folder to Project
6.  Upload artifacts/ folder to Project
7.  Select Claude Opus 4.6 model
8.  Compose activation prompt with DB connection string
9.  Send prompt, wait for Stage 1
10. Read Handoff Report, sniff-test, type confirmation
11. Repeat for Stage 2 ... Stage 5
12. Run npm install && npm test
13. Paste failures to Claude, iterate (unbounded)
14. Rename correct .env.example → .env
15. Edit .env, fill credentials
16. Run lint + typecheck + tests
17. Create feature branch, commit, push
18. Create PR, wait for team lead approval
19. Trigger GitHub Actions deploy manually
20. Coordinate with infra team for Entra ID
21. Coordinate with platform ops for SAML
```

### After: One Command

```bash
factory-run \
  --adapter aim-vue-node \
  --project ~/.factory-proof/example-project \
  --business-docs ~/.factory-proof/example-project.build-spec.yaml \
  --factory-root factory \
  --org example-org \
  --auto-approve \
  --scaffold-source ./upstream/template
```

The command:
1. Creates the project directory
2. Copies the scaffold template (`--scaffold-source`) via `cp -a`
3. Discovers and validates the adapter (`--adapter aim-vue-node`)
4. Runs pre-flight checks (PF-001 through PF-007): schema validation, capability matching, scaffold existence, agent file verification
5. Executes Phase 1 (6 process stages) with focused agents, verification gates, and durable state
6. Transitions to Phase 2: reads the frozen Build Specification, SHA-256 hashes it, generates a dynamic scaffold manifest
7. Generates data entities in FK-dependency order (topological sort)
8. Generates seed data (reference data + dev fixtures + fixture factory)
9. Generates each API operation individually with compile + test verification
10. Generates each UI page individually with compile + lint verification
11. Runs configuration and trimming passes
12. Executes a deep review pass (`EffortLevel::Deep`)
13. Performs final read-only validation (no edits allowed)
14. Reports totals: steps completed, tokens used, artifact path

Every step writes durable state. Every scaffolding step runs automated verification. Failures trigger retry with error feedback (max 3). Unrecoverable failures halt the pipeline and preserve state for `--resume`.

### CLI Flags Explained

| Flag | Purpose |
|------|---------|
| `--adapter <name>` | Select technology stack (adapter must exist in `factory/adapters/`) |
| `--project <path>` | Target project directory (created if needed) |
| `--business-docs <path>...` | One or more business requirement documents or pre-built Build Spec |
| `--factory-root <path>` | Path to `factory/` directory (defaults to `"factory"`) |
| `--org <slug>` | Organisation slug injected into Build Spec if agent omitted it |
| `--auto-approve` | Bypass interactive gate prompts (uses `AutoApproveGateHandler`) |
| `--max-turns <n>` | Maximum agentic turns per step (default 25) |
| `--resume <uuid>` | Resume a previously failed run by its run UUID |
| `--scaffold-source <path>` | Pre-populate project with a template before pipeline starts |

| Dimension | goa-software-factory (Original) | factory-run (New) |
|-----------|----------------------|-------------------|
| **Execution environment** | Claude Projects chat UI | Rust CLI binary (`factory-run`) |
| **Instruction delivery** | 100K+ tokens uploaded as files to Claude Projects | ~50-line focused prompts loaded per agent invocation |
| **Context per agent invocation** | Entire factory (44KB orchestrator + all skills + all artifacts) | 1–3K tokens (operation spec + pattern file + conventions) |
| **State storage** | Claude's context window (ephemeral) | `.factory/pipeline-state.json` (durable, resumable) |
| **Artifact passing** | Context window memory + selective-read instructions | File-based: each step reads from disk, writes to disk |
| **Operator workflow** | 20+ manual steps with folder uploads | Single CLI command |
| **Stage gates** | Operator types "Stage N looks good" in chat | `StepGateConfig::Checkpoint` with programmatic or `--auto-approve` gates |
| **Verification** | Self-assessed by same agent that generated code | Rust harness runs adapter commands (`compile`, `test`, `lint`, `type_check`) |
| **Generation granularity** | Batch: all controllers at once, all pages at once | Per-feature: one operation or one page per agent invocation |
| **Error recovery** | Restart entire pipeline (no state) | `--resume <uuid>` skips completed steps |
| **Session limit** | 5-hour compute window; operator monitors `/usage` | No session limit; pipeline runs to completion or halts |
| **Tech stack coupling** | Client-specific components, compliance rules, Vue/Express hardcoded in process | Process layer is technology-free; adapters are pluggable |
| **Adding a new tech stack** | Rewrite factory orchestrator + create new template | Create new adapter directory (manifest + agents + patterns) |
| **Credential handling** | Plaintext in Claude chat activation prompt | Not passed to AI; handled by adapter configuration step |
| **Pipeline audit trail** | `working-state.json` + stage validation logs (if written) | `.factory/runs/{uuid}/` with all artifacts, gate results, error logs, SHA-256 hashes |
| **Rule enforcement** | 12 Execution Discipline rules Claude self-enforces | Rust engine enforces sequencing; harness enforces quality gates |
| **Invariant checking** | None (invariants documented in prose) | Machine-checkable: `grep-absent`, `grep-present`, `file-exists`, `command-succeeds` |
| **Entity ordering** | Sequential by document order | Topologically sorted by FK dependencies (Kahn's algorithm) |
| **Retry policy** | "Up to 3 iterations"; Claude decides when to stop | Structured: max 3 retries per feature, error fed back to fresh agent |
| **Policy integration** | None | Scoped `PolicyBundle` with allowed commands, destructive-action blocking, token budget |
| **Concurrent adapters** | Impossible | 4 adapters ship today; adding more requires no process changes |

## 7. The Adapter System

### The Old Way: One Factory Per Stack

In goa-software-factory, the process of building a Vue + Express application was inseparable from the process of deciding what to build. The orchestrator referenced government design system web components. Stage skills referenced Express route patterns. Page-type agents contained HTML samples specific to that design system. Using a different tech stack meant rewriting the factory.

The template's `template-orchestrator.md` was an attempt to separate concerns; the factory produced a Build Specification, the template consumed it; but the separation was incomplete. The factory still contained Vue-specific page types, Express-specific API patterns, and client-specific compliance checks.

### The New Way: Pluggable Adapter Directories

An adapter is a self-contained directory that declares its capabilities and provides everything needed to generate code for one tech stack:

```
adapters/aim-vue-node/
├── manifest.yaml          # Capabilities, commands, conventions
├── agents/                # 6 focused prompts (api, ui, data, configurer, trimmer, reviewer)
├── patterns/              # 11 code generation patterns
│   ├── api/               # service, controller, route, test
│   ├── ui/                # view, state, route, test
│   ├── data/              # migration, query, validation-schema
│   └── page-types/        # 8 page-type patterns
└── validation/
    └── invariants.yaml    # 19 machine-checkable architecture rules
```

**Four adapters ship today:**

| Adapter | Stack | Capabilities |
|---------|-------|-------------|
| `aim-vue-node` | Vue 3 + Express 5 + Government Design System | Dual-stack, BFF, session auth, module system, 19 invariants |
| `next-prisma` | Next.js 15 + Prisma + Tailwind CSS | Single-stack, token auth, file uploads, 12 invariants |
| `encore-react` | Encore.ts + React Router + Drizzle ORM | Microservices, pub-sub, managed infra, 12 invariants |
| `rust-axum` | Rust Axum + SQLx + Askama + HTMX | Server-rendered, no JS framework, 12 invariants |

### Capability Matching

The engine doesn't just load an adapter; it validates that the adapter can satisfy the Build Specification's requirements. `AdapterRegistry::capabilities_match()` performs structured gap analysis:

- **Variant support**: Does the spec require `dual` but the adapter lacks `dual_stack: true`? → hard block.
- **Auth methods**: Does the spec require SAML but the adapter doesn't list it in `supported_auth`? → hard block.
- **Feature support**: Does the spec require email notifications but the adapter lacks `email_notifications: true`? → warning (degraded, not blocked).
- **Integration support**: Does the spec require audit logging but the adapter lacks `audit_logging: true`? → warning.

This check runs at pre-flight (PF-003) before any agent work begins. In the old system, a tech stack mismatch would surface as a generation error deep in Stage 4 or 5.

### Adding a New Adapter

Creating a new adapter requires:

1. Write `manifest.yaml` declaring capabilities, commands, and conventions
2. Write 5–8 agent prompt files (~50 lines each)
3. Write code pattern files with examples for the adapter's stack
4. Write `validation/invariants.yaml` with machine-checkable rules
5. Optionally provide a scaffold directory

The process layer, contract schemas, and engine code never change. The new adapter is discovered automatically by `AdapterRegistry::discover()` at startup.

## 8. Verification: From Self-Assessment to Machine Enforcement

### The Old Model: "Claude, Did You Do It Right?"

In goa-software-factory, every validation was a prompt instruction:

```markdown
## Validation Checklist
- [ ] FAC-S4-001: All endpoints in sitemap have matching controllers
- [ ] FAC-S4-002: Controller methods match HTTP verbs from API spec
- [ ] FAC-S4-003: Request/response schemas match data dictionary
...
```

Claude would read the checklist, decide whether each item passed, and report the result. The same agent that generated the code assessed whether it met the standard. This is equivalent to asking a student to grade their own exam.

The Blue Team security agent's `CLAUDE.md` listed over 20 confirmed failure modes where Claude had violated these self-assessed rules in production, including: "Hand-crafted HTML instead of running Python script", "Skipped validation entirely due to context pressure", "Reported PASS on checks that were not actually executed."

### The New Model: Machine-Enforced Gates

The verification contract (`factory/contract/schemas/verification.schema.yaml`) defines four categories of machine-enforced checks:

**Pre-Flight Gates (PF-001 through PF-007):**
Run by the Rust harness before any agent work begins.
- PF-001: Build Spec schema validation (JSON Schema)
- PF-002: Adapter manifest schema validation
- PF-003: Capability matching (variant, auth, features)
- PF-004: Scaffold directory existence
- PF-005: Required agent files exist for all declared agents
- PF-006: Required pattern files exist for all declared patterns
- PF-007: Adapter commands are executable

**Stage Gates (per-stage checks):**
Run after each process stage completes. Verify that required artifacts exist, contain expected content, and cross-reference correctly (e.g., S4-006: every entity field traces to a data model column; S4-007: enum values in API spec match data model enum definitions).

**Scaffolding Gates (per-feature verification):**
Run after every individual feature is generated. The engine calls the adapter's declared commands:
- After each data entity: `compile` + `lint`
- After each API operation: `compile` + `feature_verify`
- After each UI page: `compile` + `lint`
- After review pass: `compile` + `test` + `lint` + `type_check`
- After final validation: `compile` + `test` + `lint` + `type_check`

If a scaffolding gate fails, the error output is fed back to a fresh agent for retry (max 3 attempts).

**Architecture Invariant Checks:**
Each adapter declares machine-checkable invariants in `validation/invariants.yaml`. Five check types:

| Type | Example |
|------|---------|
| `grep-absent` | No direct `fetch()` calls in Vue views (must go through Pinia stores) |
| `grep-present` | Every route file imports the auth middleware |
| `file-exists` | `packages/shared/src/types/index.ts` exists |
| `file-absent` | No `.env` files committed (only `.env.example`) |
| `command-succeeds` | `npm run typecheck` exits 0 |

These run during final validation and can be run independently via:

```bash
factory-harness invariants \
  --project /path/to/project \
  --adapter factory/adapters/aim-vue-node
```

## 9. The Build Specification Contract

### The Central Artifact

The Build Specification is the contract between "what to build" and "how to build it." It is produced by the process layer (Stages 1–5), frozen at the phase transition, SHA-256 hashed for integrity, and consumed by the adapter layer (Stage 6).

It is deliberately **technology-free**: no SQL, no framework names, no file paths, no client-specific terminology. A Build Spec describes entities (FundingRequest, Assessment, Decision, Organization), operations (create-funding-request, approve-assessment), pages (funding-request-list, assessment-detail), business rules (state machine transitions, validation formulas, authorization policies); but never mentions Express, Vue, PostgreSQL, or design system components.

### Schema Coverage

The Build Spec schema (`build-spec.schema.yaml`) covers:

| Section | Content |
|---------|---------|
| `project` | Name, org, deployment variant (`single-public`, `single-internal`, `dual`) |
| `auth` | Audiences with methods (SAML, OIDC, API-key, basic, mock), S2S auth, session policy |
| `data_model` | Entities with typed fields (12 field types including reference), relationships, hydration hints |
| `business_rules` | State machines, validations, computations, authorization, constraints, privacy, retention |
| `api` | Resource-oriented operations with request/response shapes, audience mappings, stack targeting |
| `ui` | Pages with page types (10 variants), data sources, navigation |
| `integrations` | External system connections |
| `notifications` | Email/SMS/in-app notification templates |
| `audit` | Audit events and retention policies |
| `security` | Classification, threat model inputs |
| `traceability` | UC and TC IDs tracing through entities → operations → pages → generated code |

### Typed Rust Representation

Every section of the Build Spec has a corresponding Rust struct in `crates/factory-contracts/src/build_spec.rs`. The `Variant`, `AudienceMethod`, `FieldType`, `HttpMethod`, `StackTarget`, `PageType`, `ViewType`, and `BusinessRuleType` enums provide exhaustive, compiler-checked coverage. Parsing a Build Spec YAML into these types is a validation step; if the YAML doesn't match the schema, `serde_yaml` will produce a typed error before any agent work begins.

### What This Solves

In goa-software-factory, the "Build Specification" was a concept, not a schema. Each stage skill described what information to extract, but the format was prose-level: handoff reports, validation logs, and working-state JSON with loosely-defined fields. There was no way to validate that Stage 3's data model was consistent with Stage 4's API specification, because both were untyped documents interpreted by a single agent's context window.

With a typed Build Spec:
- Schema violations are caught at parse time, not at generation time
- Cross-reference checks (field-to-column, enum-to-enum) are computable
- The phase transition point is a hard boundary: the spec is frozen, hashed, and immutable
- Adapter selection is programmatic: capabilities are matched against spec requirements
- The same Build Spec can be consumed by any adapter; no rewriting required

## 10. Durable State and Crash Recovery

### The Problem With Ephemeral State

In the original system, a session crash meant total loss. The Operator Guide's instruction; "do not start a new project, just resume the same conversation"; was the best possible advice, because Claude's context window was the only place the pipeline state existed. If the 5-hour compute window expired, if the browser crashed, if Claude's response was interrupted, the operator waited and hoped the session could be resumed.

### Pipeline State as a Durable File

The new engine writes `.factory/pipeline-state.json` after every successful step:

```
.factory/
├── pipeline-state.json       # Current pipeline state
└── runs/
    └── {uuid}/
        ├── manifest.json     # Workflow manifest for this run
        ├── s1-business-requirements/
        │   ├── artifacts/    # Stage 1 output files
        │   └── gate-results/ # Stage 1 verification results
        ├── s6c-api-funding-requests-create/
        │   ├── artifacts/    # Generated service + controller + route + test
        │   └── verify/       # compile + feature_verify output
        ...
```

The state file tracks:
- Pipeline identity (UUID, adapter, Build Spec hash)
- Per-stage progress (status, artifact SHA-256 hashes, gate results)
- Per-feature scaffolding progress (entities/operations/pages completed, remaining, failed)
- Error log with timestamps and retry counts
- Audit trail (confirmations, rejections, manual fixes, pauses)
- Total token usage

### Resume From Checkpoint

```bash
factory-run \
  --adapter aim-vue-node \
  --project ~/.factory-proof/example-project \
  --business-docs ~/.factory-proof/example-project.build-spec.yaml \
  --factory-root factory \
  --resume a1b2c3d4-e5f6-7890-abcd-ef1234567890
```

The `--resume` flag reads the pipeline state, identifies completed steps, and builds a skip set. Completed steps are not re-executed. The pipeline resumes from the exact step where it halted.

This transforms the failure model from "crash = restart everything" to "crash = resume from last checkpoint, losing at most one step's work."

## 11. Per-Feature Generation

### The Batch Problem

The reference application demonstrates what happens when code is generated in batches: 11 controllers generated simultaneously, with no verification between them. A schema misalignment in the `audit_entry` table propagated through every controller that referenced it. A gateway proxy configuration error affected every BFF route. These errors were invisible during generation and surfaced only during post-generation debugging.

### One Feature, One Agent, One Verification

The new factory invokes a fresh agent for each feature:

**Data entities** are generated in FK-dependency order using Kahn's topological sort:
```
Entity without FKs → Entity referencing it → Entity referencing both → ...
```
Each entity generation is followed by `compile` + `lint`. If the DDL migration fails to compile, the error is fed to a fresh agent for retry before any downstream entity is attempted.

**Seed data** is generated as a dedicated step after all entities: reference data SQL, development fixtures SQL, a `run-seeds.ts` runner, and a `fixture-factory.ts` module for test data generation.

**API operations** are generated one at a time. Each invocation receives:
- The operation specification from the Build Spec (~200 tokens)
- The relevant entity fields from the data model
- The adapter's code pattern files (service, controller, route, test)
- The adapter's directory conventions

After generation, the harness runs `compile` + `feature_verify`. If the test fails, the error output is fed to a fresh agent with the original spec + the error message. This repeats up to 3 times.

**UI pages** follow the same pattern: one page per agent invocation, with `compile` + `lint` verification after each.

### Why Fresh Context Matters

In goa-software-factory, the agent that generated the 50th API endpoint had 49 endpoints' worth of accumulated context diluting its attention. The instructions for the current endpoint competed with the residue of all prior endpoints. The quality of the 50th endpoint was observably lower than the quality of the first.

In the new factory, the agent generating the 50th endpoint has the same 1–3K token context as the agent that generated the first. Quality is consistent across the entire application because context never accumulates.

## 12. Operator Experience

### From 20 Steps to One Command

The operator's role changes from "orchestrate a Claude chat session" to "run a CLI command and review the output."

| Operator Action | Before | After |
|----------------|--------|-------|
| Install factory tools | Clone repo, `git pull` before every run | Factory ships in OAP monorepo; `cargo build` once |
| Set up project | Web portal → clone → create artifacts folder | `--project` flag creates directory |
| Provide template | Manual upload to Claude Projects | `--scaffold-source` flag copies template |
| Provide business docs | Manual upload to Claude Projects | `--business-docs` flag (file paths) |
| Select tech stack | Implied by template choice | `--adapter` flag with capability validation |
| Start pipeline | Compose activation prompt in chat | Run `factory-run` command |
| Gate confirmations | Type "Stage N looks good" in chat | `--auto-approve` or interactive CLI prompt |
| Monitor progress | Watch Claude's chat output | Pipeline state file + structured progress output |
| Handle failures | Paste errors to Claude, iterate | Automatic retry with error feedback (max 3) |
| Handle crashes | Wait for compute window reset, hope session resumes | `--resume <uuid>` |
| Post-generation fixes | Open-ended debug cycle with Claude | Per-feature verification during generation prevents most issues |
| Credential security | Paste DB password in Claude chat | Credentials handled outside AI context |

### The Standalone Harness

The `factory-harness` binary provides verification capabilities independent of the full pipeline:

```bash
# Run pre-flight checks against a Build Spec and adapter
factory-harness preflight \
  --build-spec path/to/spec.yaml \
  --adapter factory/adapters/aim-vue-node

# Run stage gate checks
factory-harness gate \
  --stage s3-data-model \
  --project /path/to/project \
  --checks-file factory/contract/schemas/stage-outputs/data-model.checks.yaml

# Run per-feature verification
factory-harness feature \
  --feature-id create-funding-request \
  --type api \
  --project /path/to/project \
  --commands "npm run build" "npm test"

# Run architecture invariants
factory-harness invariants \
  --project /path/to/project \
  --adapter factory/adapters/aim-vue-node

# Initialize or inspect pipeline state
factory-harness state init
factory-harness state show
```

This allows operators to verify specific aspects of a generated project without running the full pipeline; useful for debugging, for CI integration, and for validating hand-written code against adapter invariants.

## 13. Security and Governance

### Credential Isolation

The original factory required operators to paste a PostgreSQL connection string (including password) into the Claude chat activation prompt. The Operator Guide flagged this as sensitive but offered no mitigation.

The new factory never passes credentials to AI agents. Database configuration, auth provider setup, and secret injection are handled by the adapter's `configure` agent, which reads from `.env.example` templates and writes `.env` files locally; the secrets themselves come from the operator's environment, not from the AI's context.

### Policy Integration

The engine generates a scoped `PolicyBundle` for each pipeline run:

```
factory:{adapter_name}
├── allow-adapter-commands    # Only declared commands can be executed
├── block-dangerous           # No destructive operations (rm -rf, force push, etc.)
└── token-budget (advisory)   # Max diff size scaled by project complexity
```

This integrates with OAP's `policy-kernel` crate. The adapter cannot execute arbitrary commands; only the commands declared in its `manifest.yaml` (e.g., `npm run build`, `npm test`, `npm run lint`). Destructive operations are blocked at the policy level, not by prompt instruction.

### Audit Trail

Every pipeline run produces a structured audit trail in `.factory/runs/{uuid}/`:

- All intermediate artifacts with SHA-256 content hashes
- All gate check results (pass/fail/warning with messages)
- All error logs with timestamps, retry counts, and error output
- Stage confirmations and rejections (who approved, when)
- Token usage per step and cumulative

This is a durable, inspectable record; not a chat transcript that may be ephemeral or inaccessible.

### Invariant Enforcement

Architecture invariants are not prose rules that Claude is asked to follow. They are machine-checkable assertions that run after code generation:

```yaml
# From aim-vue-node/validation/invariants.yaml
- id: INV-001
  description: No direct fetch() in Vue views; use Pinia stores
  check_type: grep-absent
  pattern: "fetch\\("
  glob: "apps/web*/src/views/**/*.vue"
  severity: error

- id: INV-005
  description: Auth middleware imported in every route file
  check_type: grep-present
  pattern: "requireAuth|optionalAuth"
  glob: "apps/api*/src/routes/**/*.ts"
  severity: error

- id: INV-012
  description: TypeScript compilation succeeds
  check_type: command-succeeds
  command: "npm run typecheck"
  severity: error
```

If INV-001 fails, the generated code has a direct `fetch()` call in a Vue view. This is caught automatically, not by hoping Claude remembered the rule.

## 14. Multi-Stack Extensibility

### The Original Constraint

goa-software-factory was built for one tech stack: Vue 3 + Express 5 + a government design system. The template was the only supported scaffold. Using a different stack required rewriting the factory; not just adding a template, but modifying the orchestrator, stage skills, page-type agents, and validation checklists.

### Four Stacks, Same Process

The new factory ships with four adapters that span fundamentally different architectures:

| Adapter | Backend | Frontend | Database | Rendering |
|---------|---------|----------|----------|-----------|
| `aim-vue-node` | Express 5 (BFF) | Vue 3 SPA + Gov Design System | Raw SQL (pg) | Client-side |
| `next-prisma` | Next.js Route Handlers | Next.js React | Prisma ORM | Hybrid SSR/CSR |
| `encore-react` | Encore.ts Microservices | React Router | Drizzle ORM | Client-side |
| `rust-axum` | Axum | Askama Templates + HTMX | SQLx | Server-rendered |

All four consume the same Build Specification. The process layer does not change. The contract schemas do not change. The engine code does not change. Only the adapter directory differs.

### What an Adapter Owns

An adapter is authoritative over:

- **Technology choices**: Which framework, which ORM, which template engine
- **Code patterns**: How a service, controller, route, or page is structured in this stack
- **Directory conventions**: Where files go (`apps/api/src/services/` vs `src/app/api/` vs `services/{name}/`)
- **Commands**: How to compile, test, lint, type-check in this stack
- **Architecture invariants**: What rules the generated code must follow
- **Agent prompts**: How to instruct Claude to generate code for this stack

An adapter has no authority over:

- **What to build**: Entities, operations, pages, business rules; all come from the Build Spec
- **Pipeline sequencing**: Stage order, gates, retry policy; all owned by the process layer
- **Verification contract**: Gate definitions, check types; all owned by the contract layer
- **Pipeline state**: Progress tracking, crash recovery; all owned by the engine

This separation means a new adapter author needs zero knowledge of the process layer, the engine code, or the contract schemas. They write a manifest, agent prompts, patterns, and invariants; and the engine handles everything else.

## 15. What Was Preserved

The new factory is not a rejection of goa-software-factory. It is a formalization of its best ideas:

### Preserved Principles

- **Stage-gated sequential pipeline.** The original 5-stage progression (requirements → service design → data model → API → UI) is preserved exactly, with a pre-flight stage added at the beginning and an adapter handoff stage at the end.

- **Technology-free requirements analysis.** Stages 1–5 produce a technology-free specification. The insight that process and implementation should be separable was present in goa-software-factory's Template Mode; it is now enforced by contract.

- **Human gates at critical transitions.** The operator still reviews and approves at stage boundaries. `--auto-approve` is an opt-in that replaces the gate, not a removal of it.

- **Self-review cycles.** goa-software-factory's API build-review-remediate cycle (api-builder → api-reviewer, max 3 iterations) is preserved in the per-feature build-test-fix loop, now with machine verification instead of self-assessment.

- **Template Mode as the default.** goa-software-factory's innovation of separating "what to build" (factory) from "how to build it" (template) is now the only mode of operation. The Build Specification *is* the template handoff.

- **Domain knowledge in the AIM template.** The 13 modules, 21 architecture invariants, variant system, and module orchestration layer from the AIM template are preserved in the `aim-vue-node` adapter's manifest, patterns, and invariants.

- **Traceability.** goa-software-factory's UC and TC IDs in the BRD are preserved in the Build Spec's `traceability` section, now with machine-checkable forward and backward references.

### What Changed

- **The context window is no longer the runtime.** A Rust engine manages sequencing, state, and verification.
- **Rules are enforced by code, not by instruction.** The 12 Execution Discipline rules are replaced by engine logic and harness checks.
- **One agent per feature, not one agent for everything.** Context is bounded and fresh.
- **Verification is external.** Real compiler, real test runner, real linter; not self-assessment.
- **State is durable.** Pipeline state survives crashes, session limits, and operator errors.
- **The adapter is pluggable.** Four stacks ship today; adding a fifth requires no process changes.
- **The operator runs a command, not a chat session.** Manual coordination is replaced by CLI flags.

### The Upstream Sync

The relationship between goa-software-factory, the AIM template, and the OAP factory is tracked by `factory/upstream-map.yaml` (spec 088). When goa-software-factory evolves; new stage skills, updated page-type specs, revised compliance checks; the changes are translated through the upstream map into the appropriate OAP layer: process-level changes go to `factory/process/`, adapter-level changes go to `factory/adapters/aim-vue-node/`, and client-specific coupling is stripped during translation.

This means goa-software-factory remains the authoritative source for client domain knowledge, while OAP's factory remains the authoritative execution architecture. Neither overwrites the other.
